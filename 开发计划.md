# 通用版本控制工具开发计划

## 1. 项目概述

本项目目标是开发一款轻量化、现代化的通用版本控制桌面工具，统一管理 Git 与 SVN 仓库，后续预留 P4 支持能力。应用前期面向 Windows 环境，采用 Tauri + Vite 技术栈，提供桌面应用、右键菜单、命令行等多种使用入口。Linux 版本暂不纳入首版开发范围，但核心架构尽量避免不必要的平台绑定，便于未来重新评估。

核心价值：

1. 降低同一台电脑上混用 Git、SVN 等版本控制工具的操作成本。
2. 提供比传统 SVN 工具更现代、轻量、直观的交互体验。
3. 将常用提交、更新、差异查看、忽略文件管理等流程尽量收敛为一键操作。
4. 提供类似 GitHub 的仓库文件浏览、代码评审与团队质量协作体验。

## 2. 建设原则

1. 轻量优先：避免把工具做成完整 IDE，只聚焦版本控制高频流程。
2. 一致体验：Git 与 SVN 尽量使用统一的信息结构、状态展示和操作入口。
3. 可扩展架构：核心版本控制能力通过适配器模式抽象，便于后续接入 P4。
4. 操作安全：关键操作需要清晰预览、确认、错误反馈和可追踪日志。
5. 本地优先：仓库信息、用户设置、监听配置等默认保存在本地。
6. 现代美观：UI 视觉风格参考 Windows 11，强调简洁、通透、层次清晰、动效克制和高可读性。

## 3. 技术方案

### 3.1 技术栈

1. 桌面框架：Tauri。
2. 前端构建：Vite。
3. 前端框架：React。
4. 后端能力：Rust 侧负责文件系统、进程调用、系统集成、右键菜单、通知等能力。
5. 命令集成：通过本机已安装的 git、svn、gh 命令执行实际版本控制操作，没有的话提供安装选项。
6. 本地存储：SQLite。

### 3.2 平台适配策略

1. 开发优先级：首版只承诺 Windows，优先保证 Windows 下的安装、启动、文件监听、右键菜单、通知、命令调用和打包体验。
2. 未来适配原则：核心逻辑、命令封装、数据模型和 UI 交互尽量保持平台无关，避免写死不必要的路径、换行符和系统集成逻辑。
3. 系统集成隔离：右键菜单、系统通知、开机启动、文件管理器打开等能力按平台封装，首版只实现 Windows 分支。
4. 命令兼容：git、svn、gh 调用统一经过命令执行层处理，首版适配 Windows PowerShell / CMD 环境。
5. 打包策略：首个可试用版本提供 Windows 安装包；Linux 构建与测试暂不安排。

### 3.3 架构分层

1. UI 层：仓库列表、状态面板、文件浏览器、文件变更列表、diff 视图、评审视图、设置页、弹窗通知等。
2. 应用服务层：仓库管理、检测调度、提交流程、更新流程、ignore 管理、文件浏览、代码评审、质量检查、日志管理。
3. VCS 适配层：GitAdapter、SvnAdapter，未来扩展 P4Adapter。
4. 系统集成层：文件监听、系统通知、右键菜单、命令行入口、本地配置。
5. 命令执行层：统一封装命令调用、输出解析、错误处理、超时处理和权限检查。

## 4. 功能范围

### 4.1 MVP 功能

MVP 版本建议优先完成以下能力：

1. 仓库添加与识别
   - 手动添加本地仓库路径。
   - 自动识别当前目录属于 Git、SVN 或两者共存。
   - 记录已添加仓库。
   - 展示仓库名称、路径、类型、当前分支或 SVN URL、最近检测时间。

2. 仓库状态检测
   - Git：检测本地修改、暂存状态、未跟踪文件、远端更新。
   - SVN：检测本地修改、新增文件、远端更新。
   - 支持手动刷新单个仓库状态。

3. 自动检测线上更新
   - 支持按配置间隔轮询，例如 1H。
   - 检测到远端更新后弹窗提示。
   - 在提示中提供拉取或更新最新代码的入口。

4. 基础提交流程
   - 展示变更文件列表。
   - 支持选择提交文件。
   - 支持填写提交信息。
   - Git 支持 add / commit / push，默认提交后 push，可在设置中更改。
   - SVN 支持自动 add 新增文件，再 commit。
   - 提供一键提交入口，并显示执行进度与结果。

5. 更新流程
   - Git 支持 pull。
   - SVN 支持 update。
   - 对存在本地修改的仓库，在更新前提示风险。

6. diff 对比
   - 展示文件变更内容。
   - 支持新增、删除、修改文件的基础 diff 查看。
   - 展示方式参考 Codex 风格，强调清晰的左右或分块对比。

7. 本地仓库文件浏览
   - 支持在界面中浏览本地仓库目录树。
   - 支持打开并预览常见文本代码文件。
   - 支持按分支或当前工作区查看文件内容。
   - 支持从文件浏览视图跳转到变更状态、diff 和 ignore 操作。

8. ignore 管理
   - Git：支持编辑 .gitignore。
   - SVN：支持编辑 svn:ignore 规则。
   - SVN：支持图形化按目录管理 ignore 规则。
   - 支持从文件变更列表中一键加入忽略。
   - 支持手动添加忽略项。

9. Windows 右键菜单
   - MVP 包含 Windows 文件夹右键菜单入口。
   - 支持从右键菜单打开工具、检测仓库、更新、提交等常用操作。

10. 多语言
   - 支持中英文界面切换。
   - 文案、菜单、通知、错误提示统一走国际化资源。

11. 设置
   - 自动检测开关。
   - 检测时间间隔。
   - 命令路径检测。
   - 默认提交行为，包括是否默认 push。
   - 通知开关。
   - 语言设置。

### 4.2 第二阶段功能

1. 命令行入口
   - 支持在终端中调用工具打开当前目录。
   - 支持基础命令，例如 status、update、commit。

2. 线上仓库文件浏览
   - Git 支持 GitHub 线上仓库文件浏览。
   - SVN 支持线上仓库目录与文件预览。
   - 文件浏览体验参考 GitHub 仓库文件页。
   - 架构预留 GitLab、Gitea、Bitbucket 等平台适配能力。

3. GitHub gh 集成
   - 检测 gh 是否安装与登录。
   - 支持查看远端仓库信息。
   - 支持一键打开仓库页面、PR 页面、Actions 页面。
   - 支持浏览线上仓库文件，体验参考 GitHub 仓库文件页。
   - 支持查看分支、提交、标签等基础线上信息。
   - 支持创建 PR、查看 PR 列表和 PR 详情。
   - 支持查看 GitHub Actions 状态。

4. 多仓库工作区
   - 支持批量刷新。
   - 支持批量更新。
   - 支持按项目或分组管理仓库。

5. 团队协作与代码评审
   - 支持围绕分支、提交或变更集创建评审任务。
   - 支持查看待评审、已通过、需修改、已合并等状态。
   - 支持在 diff 中添加行级评论。
   - 支持 code review 结论：通过、请求修改、仅评论。
   - Git 支持线上 PR 评审流程。
   - SVN 支持线上评审流程，需要通过应用侧评审任务与 SVN 服务端存储机制实现。
   - 支持团队级评审人、审批规则和质量门禁配置。

6. 质量保证流程
   - 支持提交前检查清单。
   - 内置不同语言和技术栈的格式化、lint、测试等质量检查模板。
   - 支持用户对质量检查模板进行增删改查。
   - 支持按仓库、语言、团队配置启用或禁用检查项。
   - 支持展示 GitHub Actions 或其他 CI 状态。
   - 支持把质量检查结果作为提交、推送或合并前的提示。

7. 操作历史与日志
   - 记录每次命令执行内容、时间、结果。
   - 提供失败详情，便于排查。

### 4.3 后续扩展功能

1. P4 适配器。
2. 更完整的冲突处理引导。
3. 分支、标签、stash 等进阶 Git 操作。
4. SVN 分支 / tag 辅助操作。
5. 内置文件预览与更强 diff 能力。
6. 更新检查、自动升级、插件化 VCS 支持。
7. 更完整的团队权限、审批规则和质量门禁配置。
8. 支持更多代码托管平台，例如 GitLab、Gitea、Bitbucket。

## 5. 页面与交互规划

### 5.0 UI 视觉风格

整体 UI 需要现代、美观，并贴近 Windows 11 的桌面应用气质。

设计方向：

1. 视觉语言
   - 参考 Windows 11 的 Fluent Design 风格。
   - 使用柔和背景、轻量阴影、清晰边界和适度圆角。
   - 界面应轻盈、干净，避免传统 SVN 工具的老旧和拥挤感。

2. 布局
   - 主界面采用高效工作台布局，优先支持仓库扫描、状态查看、提交、更新、diff 等高频操作。
   - 左侧仓库导航、顶部操作栏、中部内容区、底部状态栏保持稳定。
   - 信息密度要适合开发工具，不做营销式大卡片首页。

3. 控件
   - 按钮、输入框、列表、标签、状态徽标、菜单、弹窗统一设计规范。
   - 主要操作使用清晰的主按钮，危险操作使用明确的警示样式。
   - 状态类信息使用颜色和文字共同表达，避免只依赖颜色。

4. 主题
   - 首版至少完成浅色主题。
   - 预留深色主题能力，后续支持跟随系统主题。
   - 色彩以中性色为主，辅以蓝色、青绿色等清晰但不刺眼的强调色。

5. 动效
   - 使用克制的过渡动效，例如列表刷新、弹窗、状态切换。
   - 不使用影响效率的复杂动画。

6. 可用性
   - 支持高 DPI 屏幕。
   - 文本、按钮和列表项在常见窗口尺寸下不重叠、不截断关键内容。
   - 错误、空状态、加载状态和成功状态都需要有清楚反馈。

### 5.1 主界面

主界面应直接进入仓库管理和操作体验，不设计营销式首页。

建议布局：

1. 左侧：仓库列表与分组。
2. 顶部：当前仓库名称、路径、类型、刷新、更新、提交等主要操作。
3. 中部：仓库状态概览与变更文件列表。
4. 右侧或下方：文件 diff、操作详情、命令输出。
5. 底部：状态栏，显示最近检测时间、命令执行状态、错误提示。
6. 视觉：整体贴近 Windows 11 工作型应用，保持清爽、现代、易扫视。

### 5.2 仓库文件浏览

推荐能力：

1. 提供类似 GitHub 的仓库文件树和面包屑导航。
2. 支持在本地仓库和线上仓库之间切换浏览。
3. Git 支持选择分支、tag、commit 查看文件。
4. SVN 支持选择线上 URL、目录路径和 revision 查看文件。
5. 文本文件支持语法高亮、行号、复制路径、打开所在目录。
6. 二进制文件显示文件信息和可用操作，不强行预览。
7. 从文件视图可跳转到提交历史、diff、评审评论和 ignore 管理。

### 5.3 提交流程

推荐流程：

1. 自动检测当前仓库类型。
2. 拉取并展示当前变更。
3. 用户选择需要提交的文件。
4. 用户填写提交说明。
5. SVN 自动添加新增文件；Git 自动 stage 被选中的文件。
6. 执行提交。
7. Git 默认执行 push，用户可在设置中改为仅本地 commit。
8. 展示结果与失败原因。

### 5.4 远端更新提醒

推荐流程：

1. 后台按间隔检测远端更新。
2. 检测到更新后发出系统通知。
3. 用户点击通知后进入对应仓库详情页。
4. 页面展示远端更新摘要。
5. 用户选择立即更新、稍后提醒或关闭提醒。

### 5.5 代码评审与质量流程

推荐流程：

1. 用户从分支、提交、PR 或本地变更创建评审任务。
2. 系统生成变更摘要、文件列表和 diff。
3. 评审人可在文件或具体行上添加评论。
4. 评审人提交结论：通过、请求修改或仅评论。
5. 系统展示质量检查结果，例如 lint、测试、CI 状态。
6. 对未通过检查或未完成评审的变更，在提交、推送或合并前提示风险。

SVN 线上评审实现思路：

1. 由于 SVN 没有原生 PR，应用需要提供 Review Provider 抽象。
2. GitHub Provider：通过 gh / GitHub API 对接 PR、review、comment、checks。
3. SVN Provider：由应用生成评审任务、diff、评论和结论，并同步到团队共享的评审存储。
4. SVN 评审存储首选方案：在 SVN 服务端配置一个专用评审目录，例如 /_reviews，用 JSON 元数据、patch/diff 文件和评论文件保存评审信息。
5. 可选方案：后续提供独立轻量评审服务，用 HTTP API 存储 SVN 评审数据。
6. MVP 阶段优先设计 SVN 评审数据格式和同步流程，再逐步完善权限、通知和审批规则。

## 6. 数据模型初稿

### 6.1 Repository

字段建议：

1. id：仓库唯一标识。
2. name：仓库名称。
3. path：本地路径。
4. vcsType：git / svn / mixed / unknown。
5. remoteUrl：远端地址。
6. branchOrRevision：Git 分支或 SVN revision。
7. groupId：所属分组。
8. lastCheckedAt：最近检测时间。
9. autoCheckEnabled：是否自动检测。
10. checkIntervalMinutes：检测间隔。

### 6.2 FileEntry

字段建议：

1. path：文件或目录路径。
2. name：文件或目录名称。
3. type：file / directory / symlink / binary / unknown。
4. source：local / remote。
5. revision：分支、commit、tag 或 SVN revision。
6. size：文件大小。
7. lastModifiedAt：最近修改时间。

### 6.3 ChangeItem

字段建议：

1. path：文件路径。
2. status：added / modified / deleted / renamed / untracked / conflicted。
3. vcsType：git / svn。
4. selected：是否选中。
5. ignored：是否已忽略。

### 6.4 ReviewTask

字段建议：

1. id：评审任务唯一标识。
2. repositoryId：所属仓库。
3. title：评审标题。
4. sourceRef：来源分支、提交、PR 或本地变更集。
5. targetRef：目标分支或目标 revision。
6. status：draft / open / approved / changes_requested / merged / closed。
7. reviewers：评审人列表。
8. checksStatus：quality_pending / quality_passed / quality_failed。
9. createdAt：创建时间。
10. updatedAt：更新时间。

### 6.5 ReviewComment

字段建议：

1. id：评论唯一标识。
2. reviewTaskId：所属评审任务。
3. filePath：文件路径。
4. lineNumber：行号。
5. body：评论内容。
6. author：评论人。
7. status：open / resolved。
8. createdAt：创建时间。

### 6.6 QualityCheckTemplate

字段建议：

1. id：模板唯一标识。
2. name：模板名称。
3. language：语言或技术栈，例如 JavaScript、TypeScript、Rust、Python、CSharp。
4. checkType：format / lint / test / build / custom。
5. command：执行命令。
6. enabled：是否启用。
7. scope：global / team / repository。
8. editable：是否允许用户编辑。
9. createdAt：创建时间。
10. updatedAt：更新时间。

### 6.7 TeamConfig

字段建议：

1. id：团队配置唯一标识。
2. name：团队名称。
3. repositories：关联仓库列表。
4. reviewRequired：是否强制评审。
5. requiredApprovals：需要通过的人数。
6. qualityGateEnabled：是否启用质量门禁。
7. enabledQualityChecks：启用的质量检查列表。
8. reviewStorageType：github / svn_review_dir / external_service。
9. reviewStorageLocation：评审数据存储位置。

### 6.8 OperationLog

字段建议：

1. id：日志唯一标识。
2. repositoryId：所属仓库。
3. operation：status / update / commit / push / ignore / diff / browse / review / quality_check。
4. command：实际执行命令。
5. startedAt：开始时间。
6. endedAt：结束时间。
7. success：是否成功。
8. output：标准输出。
9. error：错误输出。

## 7. 命令能力规划

### 7.1 Git 命令

1. 检测仓库：git rev-parse --show-toplevel。
2. 状态：git status --porcelain。
3. 远端检测：git fetch 或 git ls-remote。
4. 更新：git pull。
5. 暂存：git add。
6. 提交：git commit。
7. 推送：git push。
8. diff：git diff。
9. ignore：编辑 .gitignore。

### 7.2 SVN 命令

1. 检测仓库：svn info。
2. 状态：svn status。
3. 远端检测：svn status -u。
4. 更新：svn update。
5. 新增文件：svn add。
6. 提交：svn commit。
7. diff：svn diff。
8. ignore：svn propset / propedit svn:ignore。
9. 线上目录浏览：svn list。
10. 线上文件预览：svn cat。
11. 指定 revision 查看：svn list -r、svn cat -r。
12. 评审数据同步：svn add、svn commit、svn update 专用评审目录。

### 7.3 gh 命令

1. 环境检测：gh auth status。
2. 仓库信息：gh repo view。
3. 打开页面：gh repo view --web。
4. 线上文件浏览：gh api repos/{owner}/{repo}/contents/{path}。
5. PR 列表与详情：gh pr list、gh pr view。
6. PR 评审：gh pr review。
7. CI 状态：gh run list、gh run view。
8. PR 创建：gh pr create。
9. 后续扩展：gh pr checkout、gh run watch。

### 7.4 质量检查命令

质量检查命令按仓库、团队、语言或技术栈配置执行。系统提供内置模板，同时支持用户增删改查、启用和禁用。

1. 格式检查：例如 npm run format:check、cargo fmt --check。
2. lint 检查：例如 npm run lint、cargo clippy。
3. 测试检查：例如 npm test、cargo test。
4. 自定义命令：用户可为不同仓库配置多个检查步骤。

## 8. 开发阶段计划

### 阶段 0：Git / SVN 基础可用闭环

目标：

## 10. 风险与应对

1. SVN 输出格式和版本差异
   - 应对：优先使用稳定命令输出，建立样例测试数据。

2. Git / SVN 命令执行耗时或卡死
   - 应对：统一加入超时、取消、进度提示和后台任务机制。

3. Windows 右键菜单实现和安装包权限差异
   - 应对：在技术预研阶段优先验证注册表、安装器权限、卸载清理和用户权限问题。

4. 自动检测频繁导致性能或网络压力
   - 应对：支持按仓库配置间隔，默认较保守，避免启动时同时检测所有仓库。

5. 一键提交存在误操作风险
   - 应对：提交前展示文件清单与操作摘要，必要时加入确认步骤。

6. 冲突处理复杂
   - 应对：MVP 只做清晰提示和跳转，后续再提供更完整的冲突解决体验。

7. 线上仓库浏览依赖平台 API、SVN 服务端能力和认证状态
   - 应对：GitHub 首版优先基于 gh 和公开 API；SVN 基于 svn list / svn cat；未登录或权限不足时提供明确引导。

8. 代码评审流程容易膨胀为完整协作平台
   - 应对：首版聚焦 PR / SVN 评审任务、行级评论、评审结论和质量检查，不做复杂项目管理。

9. SVN 线上评审缺少原生 PR 模型
   - 应对：设计 SVN Review Provider，通过专用评审目录或后续轻量服务保存评审元数据、diff、评论和结论。

10. 质量模板过多导致配置复杂
   - 应对：内置模板按语言分组，默认只启用常见检查，高级配置放入团队设置。

### 11.2 待细化问题

#### 11.2.1 SVN 线上评审数据存储方式

背景：

SVN 本身没有类似 GitHub PR 的原生评审模型，也没有标准的 review、comment、approval 数据结构。因此如果要支持 SVN 线上评审，应用必须额外设计评审数据的存储和同步方式。

可选方案：

1. SVN 专用评审目录
   - 在 SVN 仓库中创建专用目录，例如 /_reviews 或 /tools/reviews。
   - 每个评审任务保存为一个独立目录，内部包含 metadata.json、changes.diff、comments.json、checks.json 等文件。
   - 优点：不需要额外服务，所有数据随 SVN 权限和历史一起管理。
   - 缺点：评审数据会进入 SVN 仓库，需要约定目录权限和提交规范。

2. 独立轻量评审服务
   - 单独提供一个 HTTP 服务保存 SVN 评审任务、评论、审批和质量结果。
   - 优点：体验更接近 GitHub PR，可以做通知、权限、查询、统计等能力。
   - 缺点：开发和部署成本更高，不再是纯桌面工具。

3. 本地 SQLite 评审记录
   - 评审数据只保存在本机 SQLite。
   - 优点：实现最快。
   - 缺点：无法真正支持团队线上评审，不满足团队协作目标。

推荐方向：

首版推荐使用 SVN 专用评审目录，作为不依赖额外服务的团队协作方案；同时在数据结构上预留 external_service 类型，后续可迁移到独立轻量评审服务。

需要确认：

1. 是否允许在 SVN 仓库中增加专用评审目录。
2. 专用目录放在每个业务仓库内，还是统一放在一个独立 SVN 仓库中。
3. 评审数据是否允许被所有开发者读取，还是需要按团队或项目隔离权限。

#### 11.2.2 SVN 评审目录命名、权限、提交规范和冲突处理

背景：

如果使用 SVN 专用评审目录，必须定义清晰的目录结构和写入规则，否则多人同时创建评审、评论或更新状态时容易冲突。

建议目录结构：

```text
/_reviews
  /open
    /review-20260429-0001
      metadata.json
      changes.diff
      comments.json
      checks.json
  /closed
    /review-20260420-0003
      metadata.json
      changes.diff
      comments.json
      checks.json
```

建议命名规则：

1. review-{日期}-{递增序号}，例如 review-20260429-0001。
2. metadata.json 保存标题、作者、评审人、来源 revision、目标路径、状态。
3. comments.json 保存行级评论和 resolved 状态。
4. checks.json 保存质量检查结果。
5. changes.diff 保存评审对应的 diff 快照。

权限建议：

1. 普通开发者可创建评审和评论。
2. 评审人可提交审批结论。
3. 维护者可关闭、归档、修复异常评审数据。
4. 如果 SVN 服务端权限无法细分，应用层需要提供操作确认和审计日志。

冲突处理建议：

1. 每次写入评审数据前先 svn update 专用评审目录。
2. 评论追加尽量拆分为独立 comment 文件，减少多人同时编辑 comments.json 的冲突。
3. 状态变更需要检查当前评审状态，避免覆盖他人审批结论。
4. 冲突发生时优先保留所有评论内容，并提示用户手动合并。

需要确认：

1. SVN 服务端是否允许新增和保护 /_reviews 目录。
2. 是否需要在评审数据中保存真实用户名、邮箱或只保存本机账户。
3. 是否需要评审归档策略，例如关闭 30 天后移动到 closed。

#### 11.2.3 内置质量模板首批范围

背景：

质量保证流程要求内置模板，同时支持用户增删改查、启用和禁用。模板过多会增加维护成本，模板过少又无法体现价值。

建议首批模板：

1. JavaScript / TypeScript
   - npm run lint
   - npm test
   - npm run build
   - npm run format:check

2. Rust
   - cargo fmt --check
   - cargo clippy
   - cargo test
   - cargo build

3. Python
   - ruff check .
   - pytest
   - python -m compileall .

4. .NET / C#
   - dotnet format --verify-no-changes
   - dotnet test
   - dotnet build

5. 通用模板
   - 自定义命令。
   - 检查文件是否包含禁止提交的大文件。
   - 检查提交说明是否为空。

模板配置建议：

1. 模板默认禁用，由用户或团队按仓库启用。
2. 每个模板支持工作目录、命令、超时时间、失败是否阻断提交。
3. 支持按语言自动推荐，但不自动强制启用。
4. 用户可复制内置模板生成自定义模板。

需要确认：

1. 首批是否需要覆盖 Java、C++、Go、PHP 等语言。
2. 质量检查失败时是只提示，还是阻止提交 / push。
3. 团队级质量门禁是否允许个人临时跳过。

#### 11.2.4 团队级配置的存储与同步

背景：

团队级配置包括评审人、审批规则、质量门禁、模板启用状态等。如果只存在本机，则团队成员规则不一致；如果同步到仓库，则需要处理权限和冲突。

可选方案：

1. 本地 SQLite
   - 优点：实现简单，不影响仓库。
   - 缺点：每个成员配置可能不同，团队规则难以统一。

2. 随仓库同步
   - Git 仓库可使用 .gvmt/team-config.json。
   - SVN 仓库可使用 /_reviews/team-config.json 或专用配置目录。
   - 优点：团队成员共享同一份规则。
   - 缺点：配置文件会进入仓库，需要权限和变更审查。

3. 远端评审服务
   - 优点：权限、审计、通知最好做。
   - 缺点：超出纯桌面工具范围。

推荐方向：

首版采用“本地 SQLite + 可选仓库同步”的混合方案。本地保存用户个人偏好，团队规则可选择同步到仓库配置文件。

建议拆分：

1. 个人配置：语言、主题、默认 push、通知偏好，保存在本地 SQLite。
2. 仓库配置：质量模板启用状态、默认评审人、检查命令，优先支持同步到仓库。
3. 团队配置：审批人数、质量门禁、评审存储位置，可同步到仓库或 SVN 评审目录。

需要确认：

1. 团队配置变更是否需要评审。
2. 配置文件是否允许提交到业务仓库。
3. 多个团队共用一个仓库时，配置如何分组。

#### 11.2.5 中英文文案覆盖范围

背景：

已确认支持中英文界面切换。问题在于首版是否必须做到完整双语覆盖，还是先覆盖主流程。

可选方案：

1. 完整覆盖
   - 所有菜单、按钮、提示、错误信息、设置项、通知都提供中英文。
   - 优点：国际化质量高。
   - 缺点：首版文案维护成本更高。

2. 主流程覆盖
   - 仓库列表、提交、更新、diff、设置、通知等主流程提供中英文。
   - 部分低频错误或日志先保留中文或英文。
   - 优点：首版更快。
   - 缺点：体验不完全统一。

3. 架构先行，文案逐步补齐
   - 所有 UI 文案都走 i18n key。
   - 首版至少中文完整，英文逐步补齐。
   - 优点：技术债较少。
   - 缺点：首版英文体验有限。

推荐方向：

采用“架构先行 + 主流程完整双语”。所有新增文案必须走 i18n key，首版完整覆盖主流程，低频错误信息可逐步补齐。

需要确认：

1. 默认语言是中文、英文，还是跟随系统。
2. 错误日志是否需要翻译，还是保留原始命令输出。
3. 发布说明和使用文档是否也需要双语。

5. 第 5 周：GitHub / SVN 线上文件浏览、gh PR / Actions 基础集成。
6. 第 6 周：Windows 右键菜单、命令行入口、Windows 安装包预研。
7. 第 7 周：GitHub PR 评审、SVN 线上评审 Provider 初版、行级评论。
8. 第 8 周：质量检查内置模板、模板增删改查、团队级配置。
9. 第 9 周：Windows 测试、UI 视觉验收、打包、体验优化、发布试用版。

实际周期需要根据 UI 精细度、SVN 线上评审复杂度、Windows 系统集成细节和真实团队使用场景调整。

## 13. 开发时间预估

以下预估以 1 名全职开发人员依靠 Codex 辅助开发为主要基准，首版只考虑 Windows。Codex 可明显提升项目搭建、样板代码、常规功能实现、重构、测试补齐和文档维护效率，但需求取舍、产品验收、真实 SVN / GitHub / Windows 系统集成验证仍需要开发者主导。

### 13.1 MVP 版本

目标范围：

1. Windows 桌面应用可用。
2. Git / SVN 仓库添加、识别、状态检测。
3. 本地仓库文件浏览。
4. 基础 diff。
5. Git / SVN 更新与一键提交流程。
6. ignore 管理。
7. 自动检测远端更新与通知。
8. 基础设置页。
9. Windows 右键菜单。
10. 中英文界面基础切换。
11. 现代化 Win11 风格基础 UI。

预估时间：

1. 1 名开发 + Codex：5 到 7 周。
2. 1 名开发纯人工保守参考：7 到 10 周。

适合目标：先做出可实际使用的轻量版本，验证核心工作流是否顺手。

### 13.2 首个可试用版本

在 MVP 基础上增加：

1. gh 基础集成。
2. 线上 GitHub / SVN 仓库文件浏览。
3. PR 创建、PR 查看和 Actions 状态。
4. 命令行入口。
5. 操作日志。
6. 更完整的错误提示。
7. Windows 安装包。
8. SVN 线上评审 Provider 初版。
9. 质量检查内置模板与模板管理初版。

预估时间：

1. 1 名开发 + Codex：8 到 11 周。
2. 1 名开发纯人工保守参考：11 到 15 周。

适合目标：发布给小范围用户试用，收集真实仓库场景反馈。

### 13.3 较完整版本

在首个可试用版本基础上增加：

1. 团队代码评审流程。
2. 行级评论。
3. PR 状态关联。
4. GitHub Actions / CI 状态展示。
5. 本地质量检查命令配置。
6. 评审结论流转。
7. SVN 线上评审完善。
8. 团队级评审规则和质量门禁。
9. 更完整的测试覆盖和异常场景处理。

预估时间：

1. 1 名开发 + Codex：12 到 16 周。
2. 1 名开发纯人工保守参考：16 到 22 周。

适合目标：作为团队日常版本控制和质量协作工具长期使用。

### 13.4 人力影响

1. 1 名全职开发 + Codex：MVP 约 5 到 7 周，首个可试用版本约 8 到 11 周，较完整版本约 12 到 16 周。
2. 1 名全职开发纯人工：MVP 约 7 到 10 周，首个可试用版本约 11 到 15 周，较完整版本约 16 到 22 周。
3. 2 名开发协作 + Codex：若前后端、Tauri/Rust、测试发布分工清晰，首个可试用版本可压缩到 7 到 9 周。
4. 1 名兼职开发 + Codex：周期通常需要乘以 1.3 到 1.7 倍。

### 13.5 Codex 可加速的工作

1. Tauri + Vite 项目骨架搭建。
2. 前端页面、组件、状态管理和基础交互实现。
3. Git / SVN / gh 命令封装和输出解析初稿。
4. 数据模型、接口类型和本地存储逻辑。
5. diff、文件树、设置页、操作日志等常规功能。
6. 单元测试、集成测试样例和测试数据补齐。
7. 文档、发布说明、已知问题清单维护。

### 13.6 Codex 难以完全替代的工作

1. 产品范围取舍和体验验收。
2. 真实 Windows 右键菜单、通知、安装包验证。
3. 真实 SVN 服务器、认证、权限、冲突场景验证。
4. GitHub gh 登录状态、组织权限、PR / Actions 权限验证。
5. SVN 线上评审存储方案和团队权限验证。
6. 最终发布质量判断。

### 13.7 主要不确定因素

1. SVN 场景复杂度，包括认证方式、ignore 管理、远端检测和新增文件自动 add。
2. Windows 右键菜单和安装包体验要求。
3. SVN 线上仓库浏览与认证方式。
4. SVN 线上评审是使用专用 SVN 目录，还是使用独立轻量服务。
5. 团队级配置和质量模板首批覆盖范围。
